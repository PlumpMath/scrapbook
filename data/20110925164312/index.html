Notes

Note: emacs's keybinding changes slightly depending on the current major mode. This page gives the general idea. You can call “describe-key” to find the command name of any key combo input.

If some symbols are showing up as blank, see: Unicode Character Shows Blank, Question Mark, Gibberish. Best unicode support is Google Chrome.

    To what extent are uppercase letters necessary in operating systems and glyph hierarchies? For example, in the english language, capital letters are used at the start of a sentence. But this isn't necessary in naming files, since file names are most often words, not sentences. Why haven't people just done away with naming files, variables, programs with capital letters? It would making key bindings much simpler (the shift pinky problem, the caps lock problem). 
    Clearly single key bindings are better than two-key bindings, or modes such as in vi. The problem is in the global context (i.e we need more switches to have single-key bindings). You rarely see emacs functions called, for example, "Draw_Some_Parabola_x->y" , because its annoying to type. A better name would be "draw_some_parabola_xy". Binaries in unix aren't name with capital letters, color codes don't use capitals. Yet there are some cases where capitals are still used but not necessary. For example, X window config files use it (.Xdefault and Xauthority are confi files in $HOME,  .R is the config folder for the R math environment in $HOME).

    Next one would wonder if it is possible to get rid of double key bindings for other symbols, as it seems quite possible for uppercase letters. For example, why haven't people agreed to use - instead of _ in naming files, variables, functions, etc? Of the two symbols, one is more common, and people are free to put more important on either of the two. Why haven't people agreed to such a convention? 

    It seems that people have not been able to organise themselves. One could argue that an operating system is such a large structure that a consensus is impossible to achieve. If this is the case, why haven't people agreed to such a consensus on smaller structures?

    How possible is it to agree on a single type of bracket in a programming language? It seems the more complex a language becomes, the more symbols it will require. Is this true? I can think of reasons why this is not the case. For example, there are no obvious reasons why a function would need to separate two parameter fields by two sets of brackets, or if two parameter fields are required. For example, look at a traditional C-style function;

    function (parameters) { arguements} can be written as function (parameters) (arguements), because
    1) the parameter field is always required, and only one exists per function (multiple parameters are are passed into the same field)
    2) the arguements field is always required, and only one exists per function
    3) the parameters field is always follwed by the arguements field
