GNU Screen v.s. tmux

Posted by zt

4

虽说这篇 blog 标题是 screen vs tmux，但是并不局限于此。不少朋友对这两个东西还可能比较陌生，这里简单的做个广告。

很多互联网开发的程序员都是通过 ssh 连接到远程主机，并在远程主机上进行开发和执行任务，但是一个 ssh 连接往往不能很方便的在多个任务之间切换。最原始的切换方式莫过于 bash 自带的 job 管理，但这有一定的缺陷：

  • 比如你一边在写程序，突然一个同事跑过来对你说希望看看某个地方的问题如何解决
  • 比如你使用了有线网，然后同事叫你去开会，你必须切换到无线网上，手头的事情中断之后难以继续

有的人使用 emacs 能 cover 第一种情形，但是对于后面一种情况还是有点无能为力（尽管可以使用 emacs 的 server 来实现类似的效果！）。screen 和 tmux 的解决方案在于能从某个 bash 会话中产生一个他们呢自己管理的 session，这个 session 里面的程序不受到 SIGHANGUP 的影响，就算用户突然短线，运行中的程序也不会收到此 signal（默认行为是 abort），同时他们还会在这个 session 中通过一些自己的命令（和 key binding）帮助你创建新的 shell，将窗口进行分割组织，甚至共享。

说到这些功能，鼻祖自然是 GNU screen，在成为 FSF 的产品之后加入了不少新特性，虽然说 tmux 是 screen 的重写，但新增了多少功能，我看也不是那么有质的区别。screen 曾经有个  frontend 叫 byobu，现在 byobu 也开始支持 tmux 了。其实这三个玩意功能现在都差不多，前面 leemars 同学提供了一些比较(这里还有一份 key binding 的比较)，我擅自加上一些我的 comment 吧：

  • 配置能力，其实没觉得 screen 比 tmux 弱多少哈，不过 tmux 的确要更加细致一些的感觉，screen 很多时候要依靠 termcap/terminfo 来搞一些不知道什么的玩意，tmux 这点上似乎不需要这么操心
  • 但是 tmux 的确很蛋疼的招惹了 Emacs 用户，因为它的 window 是 emacs 的 buffer，而它的 panel 是 emacs 的 window… 怎么这么绕呢！
  • 竖分屏，其实 screen 也支持的哈，split -v（默认的 binding 是 C-a |）
  • 文档，screen 沿袭了 FSF 风格的文档，我喜欢；其他两个似乎只有 man page。
  • 不知道为啥进入 copy-mode 之后不能把 key binding 弄得和 vi 或者 emacs 一样，人家 bash 都考虑到了…
  • tmux 似乎还有 monitor 某个窗口是否有变化的功能，这个不错…

可能 tmux 的确是值得迁移的哈，基本可以配置成 screen 类似的 key binding，就是每次都要写 attach 比较蛋疼…

最后提一句的是 xpra 这个东西。这类似一个 X-server + screen 的作用，我们知道很多 X11 的程序必须 attach 到某个 X server 上才能执行，这导致如果我们的 X-server 必须重启，或者我们想把一个 X client 从远程的显示转移到本地，都是不可能做到的。VNC 提供了一个重量级的解决方案，它的想法是完全重新打开一个 X11 session，包括对应的 Window Manager，当然现在有不少轻量级的 WM 比如我比较喜欢的 openbox、WindowMaker、lxde，但是这样一来所有的 X clients 必须都放在一个窗口里面，这个窗口就是 vncviewer 被当前 window manager 处理产生的。xpra 是一个轻量级的解决方案，它并不需要额外设置一个完整的 X11 session，相反启动了 xpra 之后，通过设置 DISPLAY 将 X client 放到其中执行后，我们可以使用 xpra attach 连接到 xpra 的会话里面，使用执行 xpra 所在环境的 X server 和 window manager 显示这个 X client。与 screen 等类似，我们只需要 Ctrl-c 中断 xpra 就 detach 了这个 session。有了这等神器，我们是否还需要原先 X11rdp 的解决方案呢？（尝试在 RHEL5 上编译一个失败了，nnd 原因居然是 xutil 太老，attach 不到 server 上去，WTF！）
